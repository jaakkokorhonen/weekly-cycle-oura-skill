weekly-cycle-oura-skill

Data-pipeline arkkitehtuuri · v0.1 MVP

4-vaiheinen pipeline

Jokainen vaihe on itsenäinen moduuli ilman sivuvaikutuksia edelliseen — mahdollistaa yksikkötestauksen ja A/B-koeasetelman.

Vaihe 1
Ingest
Hakee Oura API v2 -vastaukset (sleep, readiness, activity, heartrate) + käyttäjän tapahtumaloki. Kirjoittaa data/raw_inputs/YYYY-MM-DD.json.
›
Vaihe 2
Normalize
Pakottaa kaikki aikaleimant UTC:ksi (EET/EEST off-by-one-riski). Validoi schema/features.v1.yaml vasten.
›
Vaihe 3
Feature Engineering
Laskee derived_*-muuttujat: HRV-delta-%, RHR vs. 30d. Duaalinen nimiavaruus A/B-vertailua varten.
›
Vaihe 4
Classify & Output
Puhdas funktio: feature-dict → luokka (HIGH_LOAD / INTEGRATION / BASELINE_DAY) + tila + tekstisuositus.

Feature-nimiavaruudet

Duaalinen nimiavaruus mahdollistaa A/B-vertailun: kumpi ennustaa paremmin seuraavan päivän suorituskykyä?

oura_* — raaka API-data
oura_hrv_avg oura_rhr oura_sleep_score oura_readiness_score oura_total_sleep_h oura_deep_sleep_min
derived_* — johdetut muuttujat
derived_hrv_delta_pct derived_rhr_vs_30d derived_sleep_debt_7d derived_caffeine_lag_h derived_alcohol_units_prev derived_resilience_trend

Avainlaskenta

# HRV-delta prosenttina 14 päivän mediaanista
derived_hrv_delta_pct = (nightly_hrv − 14d_median) / 14d_median

# RHR suhteessa 30 päivän liukuvaan keskiarvoon
derived_rhr_vs_30d    = (nightly_rhr − 30d_mean) / 30d_mean

# Trendisignaali vaatii ≥3 peräkkäistä yötä samaan suuntaan
derived_resilience_trend ∈ {EXPANDING, STABLE, DECLINING, INSUFFICIENT}

Signaalihierarkia

Yksittäinen poikkeava arvo on kohinaa, ei signaali — tämä on N-of-1-metodologian peruspiste.

Trendi (14–30 pv)
  • HRV-baseline 14 pv ikkunasta
  • RHR-trendi 30 pv liukuva
  • Resilience-taso ≥3 yötä
  • Oikeuttaa kapasiteettipäätökset
Taktinen (1–3 yötä)
  • Uni <6.5h + HRV-lasku ≥10%
  • → päiväuniehdotus
  • Kofeiini <5h ennen unta
  • → varoitus seuraavalle yölle
Riittämätön data
  • <7 yötä dataa
  • Puuttuvat tapahtumat
  • Palauttaa INSUFFICIENT
  • Ei suosituksia
